# WP-5 (BinaryTree) handoff — `allocate_shared` spine + Option A knot layer

Scope note: this handoff covers the `BinaryTree`/`BinaryTreeF` half of WP-5 only.
The `ExprF` half was completed separately and is recorded in `HANDOFF-WP5-expr.md`; `expression.hpp` and its test were not touched by this WP.

## Landed

- `include/beman/tree_algorithms/binary_tree.hpp`
  - `binary_tree_default_allocator = std::allocator<std::byte>` — a fixed (value-type-independent) default, mirroring `expr_default_allocator` / `rose_default_allocator`.
  - `BinaryTreeF<T, A, Allocator = binary_tree_default_allocator>` — gained the `Allocator` parameter (Option A / DEV-A04); fields are `child_slot_t<A, Allocator>`. With the default allocator this is bit-for-bit the old `BinaryTreeF<T, A>` — same type, same size, same cost.
  - `BinaryTreeLayer<T, Allocator = binary_tree_default_allocator>` — binder now takes both axes; `BinaryTreeLayer<T>::template F` is unchanged for callers that only ever named `T`.
  - `BinaryTreeFix<T>` — unchanged spelling and type (always the default allocator).
  - `BinaryTreeFFunctorImpl`/`Map` and `functor_typeclass<BinaryTreeF<T, A, Allocator>>` — generalized over `Allocator`, registered once for the default and any pmr instantiation.
  - `BinaryTreeLayerFoldMap` and `layer_fold_typeclass<BinaryTreeF<T, A, Allocator>>` — generalized the same way.
  - `binary_tree_element<BinaryTreeF<T, A, Allocator>>` — the element-extraction trait, updated to the 3-parameter shape.
  - `BinaryTree<T>` (D-A5): three new allocator-extended static factories, tag-first:
    - `leaf(std::allocator_arg_t, const Allocator&, T)` — accepts and discards the allocator (a leaf allocates nothing: no children to wrap in a `shared_ptr`); no allocator is stored (no stored allocator member is added to `BinaryTree` — D-A5).
    - `node(std::allocator_arg_t, const Allocator&, T, BinaryTree, BinaryTree)` — builds via the tagged `make_ptr` below, so both children's control blocks come from the resource.
    - `make_ptr(std::allocator_arg_t, const Allocator&, BinaryTree) -> std::shared_ptr<BinaryTree>` — `std::allocate_shared<BinaryTree>(alloc, std::move(tree))`.
    - `from_children_ptrs` got **no** tagged overload — see Deviations/rationale below; it takes already-built pointers and allocates nothing itself.
  - `to_fix(std::allocator_arg_t, const Allocator&, const BinaryTree<T>&) -> Fix<BinaryTreeLayer<T, Allocator>::template F>` — recurses through itself, threading `alloc` into the WP-2 tagged `make_slot` at every knot Box.
  - `from_fix(std::allocator_arg_t, const Allocator&, const Fix<F>&) -> BinaryTree<T>` — same `fold_fix` as the untagged overload (no allocator overload needed on the verb, per D-A2/Decision 9), but the rebuild algebra calls the tagged `make_ptr` for every reconstructed child.
- `tests/beman/tree_algorithms/binary_tree.test.cpp`
  - All 9 prior `TEST_CASE`s and every prior `static_assert` unchanged and green.
  - 5 new additivity `static_assert`s pinning `BinaryTreeF<T,A>` / `child_slot_t<IntFixed>` / `BinaryTreeFix<int>` to their 3-parameter/default-allocator equivalents.
  - A local `tracking_alloc<T>` / `alloc_stats` counting allocator (default-constructible, unlike box.test.cpp's — a disengaged `child_slot_t<A, Allocator>{}` must default-construct its Box's allocator, exactly as a default-constructed `pmr::polymorphic_allocator` would).
  - 3 new `TEST_CASE`s (9 → 12): `AllocatorTaggedLeafNodeAndMakePtr` (direct factory allocation counts: 0/2/1), `AllocatorTaggedToFixFromFixRouteThroughTheSuppliedAllocator` (tracking-allocator round trip: 4 knot-Box allocations proven by count *and* by `get_allocator().id` identity on the root and one grandchild level, then 4 `allocate_shared` calls on a *separate* tracking allocator for `from_fix`, then balance-checked to 0 live after `fixed` goes out of scope), `PmrToFixFromFixRouteThroughTheResource` (the plan's explicitly requested pmr null-upstream proof: `monotonic_buffer_resource` over `null_memory_resource()`, `get_allocator().resource() == &pool` on `to_fix`'s knot Boxes, completed `from_fix` round trip as the shared_ptr-side proof).

No new header was added (`binary_tree.hpp` is already in the umbrella `tree_algorithms.hpp`/`tree_algorithms.cppm`), so no module-interface or CMake edit was needed — confirmed by inspection of `tree_algorithms.cppm` (DEV-A02 precedent) and by the module-mapper-driven scan succeeding in `make compile` (the Asan config already builds through the C++20-modules dependency-scan path).

## Interface decided

- **`to_fix` allocator side (Fix side):**
  `to_fix(std::allocator_arg_t, const Allocator& alloc, const BinaryTree<T>& tree) -> Fix<BinaryTreeLayer<T, Allocator>::template F>`.
  Tag-first, per Decision 9. Recurses through itself (not through the untagged `to_fix`), so every level's knot Box — not just the root's — is threaded through `alloc`.
  Pmr binary-tree Fix type for WP-7: `Fix<BinaryTreeLayer<T, std::pmr::polymorphic_allocator<std::byte>>::template F>` — WP-7 should alias this as `pmr::BinaryTreeFix<T>`.
- **`from_fix` allocator side (rebuilt shared_ptr side):**
  `from_fix(std::allocator_arg_t, const Allocator& alloc, const Fix<F>& fixed) -> BinaryTree<T>` (`T` recovered via `binary_tree_element`, same non-deduced-context trick as the untagged overload).
  Uses the **existing** `fold_fix` (lookup-based, no allocator overload — confirmed, matching WP-4/WP-5-expr) with an algebra that calls the tagged `make_ptr` for every engaged child.
- **`BinaryTree<T>` D-A5 factories:**
  `leaf(std::allocator_arg, alloc, value)`, `node(std::allocator_arg, alloc, value, left, right)`, `make_ptr(std::allocator_arg, alloc, tree) -> std::shared_ptr<BinaryTree>` — all tag-first, all templated on `Allocator`, none add a stored allocator member (the `shared_ptr` control block remembers it, per D-A5).
- **`BinaryTreeF<T, A, Allocator = binary_tree_default_allocator>`** — Option A, exactly the ExprF/RoseF pattern: parameterize the Box-at-knot layer, bind before `Fix` via `BinaryTreeLayer<T, Allocator>::template F`, register `functor_typeclass`/`layer_fold_typeclass` generically on the 3-parameter shape.
- **Verbs unchanged**: no allocator overload on `fold_fix`/`unfold_fix`/`refold`. Confirmed — same as WP-4/WP-5-expr. The allocator rides in `to_fix`/`from_fix`/the `BinaryTree` factories, not in a verb parameter or in an allocator-carrying `fmap`: unlike `ExprF`/`RoseF`, this WP's allocator-aware *build* path is `to_fix` (a direct recursive conversion, not `unfold_fix`), so no `binary_tree_fmap(alloc)` analogous to `expr_fmap`/`rose_fmap` was needed or added. `BinaryTreeFFunctorImpl::fmap` stays the plain (default-`make_slot`) primitive; the same restriction documented for `ExprF` applies if a future WP wants to `unfold_fix` a pmr `BinaryTreeFix` directly from a coalgebra — that path is not exercised or promised here.

## Deviations

None from the plan's WP-5 (`BinaryTree` half) scope, files list, or gate.

One implementation choice worth recording, in the same spirit as WP-4's fmap-policy note (a decision the plan left open, not a departure):

- **`leaf()` and `from_children_ptrs()` got no "real" allocator-tagged treatment.** `leaf(std::allocator_arg, alloc, value)` was added per the plan's explicit example, but it allocates nothing (no children to wrap in a `shared_ptr`) so it just accepts-and-discards `alloc` — added for uniform tag-first spelling with `node()`, not because it needed one. `from_children_ptrs` got **no** tagged overload at all: it assembles a value from already-built child `shared_ptr`s, so whatever allocator built those (e.g. the tagged `make_ptr`) already did the only allocation this function could route, and there is nothing left for a tag to change. This reads the plan's "and allocator-tagged forms of `from_children_ptrs` / `make_ptr` as needed" literally: `make_ptr` needed one (it allocates); `from_children_ptrs` did not.
- **No `binary_tree_fmap(alloc)` analogue to `expr_fmap`/`rose_fmap`.** `ExprF`/`RoseF` needed an allocator-carrying fmap because their allocator-aware build path is `unfold_fix` (growing a tree from a coalgebra seed). `BinaryTreeF`'s allocator-aware build path is `to_fix` — a direct recursive conversion from an *existing* `BinaryTree<T>`, not a coalgebra-driven unfold — so threading `alloc` directly through `to_fix`'s own recursion was sufficient and simpler; no fmap needed to carry it. This mirrors `BinaryTreeFFunctorImpl::fmap` staying the plain, default-`make_slot` primitive, exactly as `ExprFFunctorImpl`/`RoseFFunctorImpl` do for their own fold-only correctness. A future WP that wants `unfold_fix<BinaryTreeLayer<T,PA>::template F>(coalgebra, ..., seed)` to build a pmr `BinaryTreeFix` from a fresh seed (rather than converting an existing tree) would need to add one, following the `expr_fmap`/`rose_fmap` template exactly — the `BinaryTreeF` generalization already in place (Option A, `Allocator` deduced in the `functor_typeclass` registration) is what such a fmap would sit on top of.

## Open questions / warnings for downstream

- **Lookup-based fold/fmap over a pmr `BinaryTreeF` layer works out of the box** (same as `RoseF`, unlike `ExprF`'s unfold restriction): the generalized `functor_typeclass`/`layer_fold_typeclass` registrations cover `fold_fix` (both APIs) and `refold`'s evaluated pass identically for the default and any pmr instantiation, *as long as the algebra's parameter type is written generically* (`const F<Result>&` using the deduced base-functor alias, not the hardcoded 2-argument `BinaryTreeF<T, Result>` alias) — this WP's own `from_fix` rebuild algebra is written this way for exactly that reason. A hand-written algebra typed against the 2-argument alias (like this test file's pre-existing `show_algebra`/`subtract_algebra`) will **not** accept a pmr-tagged intermediate layer; this bit the sub-agent once while drafting the tracking-allocator test (an attempt to fold the alloc-tagged `to_fix` result directly with `show_algebra` failed to compile) and was fixed by folding only the round-tripped default-allocator tree instead. WP-7/WP-8 should state this plainly for anyone writing a generic pmr-aware algebra against `BinaryTreeF`.
- **Unfolding a stateful (pmr) `BinaryTreeFix` from a coalgebra seed is not supported by anything landed here** — only `to_fix` (converting an existing `BinaryTree<T>`) is allocator-aware on the build side. See the Deviations entry above for what a `binary_tree_fmap(alloc)` would need to look like if a downstream WP wants that path.
- **`from_fix`'s shared_ptr-side allocator is not independently queryable** after the fact (no public API exposes an `allocate_shared`-built `shared_ptr`'s allocator), so its pmr proof is "the round trip completed without the null upstream throwing," not a direct `get_allocator()` assertion — consistent with D-A5's "no per-node storage" trade and with the same limitation already documented in WP-4/WP-5-expr for their own non-queryable positions.
- Production Box-at-knot / `shared_ptr`-spine representations are now fully enumerated: expression (Box-at-knot, WP-5-expr), rose (vector-children, WP-4), `BinaryTree` (shared_ptr-spine + Box-at-knot Fix side, this WP). Whatever WP-6 landed for the fringe tree (already present in this working tree as of this handoff, per its own `HANDOFF-WP6.md`) completes the representation matrix for WP-7.
